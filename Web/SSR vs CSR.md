# SSR vs CSR 정리

## SSR (Server-Side Rendering)

> **서버 사이드 렌더링(SSR)**은 사용자가 페이지를 요청하면, 서버에서 해당 페이지의 **완전한 HTML을 생성해 브라우저로 전달**하는 방식이다.  
> 즉, 브라우저가 HTML을 받는 시점에는 이미 화면에 보여질 데이터가 포함되어 있다.

### 장점
- **빠른 초기 렌더링 (FCP)**  
  서버에서 미리 렌더링된 HTML을 보내기 때문에, 사용자는 화면이 더 빨리 표시되는 경험을 얻는다.
- **JS 의존도 감소 → 빠른 상호작용 시점 (TTI)**  
  클라이언트에서 별도로 무거운 JS 번들을 해석할 필요가 줄어들어 초기 반응성이 좋다.
- **SEO 친화적**  
  HTML에 실제 콘텐츠가 포함되어 있어, 검색 엔진이 쉽게 인덱싱할 수 있다.

### 단점
- **높은 서버 부담**  
  요청마다 서버가 페이지를 새로 렌더링해야 하므로 트래픽이 많으면 서버 부하가 커진다.
- **TTFB(Time to First Byte)** 지연  
  서버에서 HTML을 생성하는 과정이 필요하므로, 첫 번째 바이트를 받기까지의 시간이 늘어날 수 있다.
- **전체 새로고침 발생**  
  페이지 이동 시마다 서버에 요청을 보내야 하므로, 화면이 깜빡이거나 새로고침된다.
- **사용자 인터랙션이 많은 웹앱에는 비효율적**  
  모든 요청이 서버로 가기 때문에 실시간 반응형 UI나 대화형 앱에서는 비효율적일 수 있다.

## CSR (Client-Side Rendering)

> **클라이언트 사이드 렌더링(CSR)**은 서버가 HTML 껍데기를 보내고, **브라우저가 JS를 실행하여 페이지를 직접 렌더링하는 방식**이다.  
> 이 방식은 주로 **SPA(Single Page Application)** 형태의 웹앱에서 사용된다.

### 장점
- **빠른 화면 전환과 부드러운 인터랙션**  
  페이지 이동 시 전체 새로고침 없이 필요한 데이터만 불러와 화면을 갱신한다.
- **서버 부하 감소**  
  서버는 단순히 JSON 데이터만 응답하면 되므로, SSR보다 리소스 소모가 적다.
- **앱 같은 사용자 경험**  
  화면 전환이 빠르고 부드러워, 네이티브 앱과 유사한 UX를 제공한다.

### 단점
- **초기 로딩 속도 느림 (TTI 지연)**  
  처음 페이지 진입 시 JS 번들을 모두 다운로드하고 실행해야 하므로, 초기 구동이 느릴 수 있다.
- **SEO 취약**  
  초기 HTML이 비어 있고 JS 실행 후에야 콘텐츠가 렌더링되므로, 검색 엔진이 내용을 인식하지 못할 수 있다.
- **JS 의존도 높음**  
  브라우저 성능에 따라 렌더링 속도나 UX 품질이 달라질 수 있다.
